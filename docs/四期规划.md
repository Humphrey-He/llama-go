## llama-go：增量推理与 KV Cache 闭环重构 — 开发文档（v1.0 可执行版）
日期：2026-03-27

---

## 一、项目概述（目标摘要）
目标：将现有 Demo 级推理链路改造为真实可用的增量推理闭环，支持 prefill + autoregressive decode、past_key_values 复用与多轮会话，KV Cache 必须参与实际计算并带来第二轮推理性能提升（≥30% TTFT 降低）。

非目标（本轮不做）：纯 Go 推理、分布式调度、GPU 多卡、量化优化、前端 UI。

---

## 二、验收核心能力（P0）
- 支持 prefill + decode（autoregressive）
- 支持 past_key_values 正确复用（全部层）
- 支持多轮对话（session_id 管理）
- KV Cache 真正参与计算，第二轮推理显著加速
- 并发 ≥50 session 且不串 session、不污染 cache
- No JSON 序列化 tensor；Go 仅传 session_id，KV 存于 Python 内存

---

## 三、总体架构（改造后）
控制面（Go API 层，session 管理、cache metadata）↔ 数据面（Python backend，推理 + KV Cache）↔ Transformer 模型

架构要点：
- Go 负责 session lifecycle、路由、观测埋点与并发控制
- Python 负责 tokenize、prefill、incremental decode、past_key_values 存储与管理
- Go 与 Python 交互仅通过 session_id +请求参数（不传 tensor）

---

## 四、接口协议（必须严格实现）

POST /api/generate

请求示例
```json
{
  "session_id": "string（可选）",
  "prompt": "string",
  "max_new_tokens": 128,
  "temperature": 0.7,
  "top_p": 0.9,
  "top_k": 50,
  "stream": false
}
```

响应示例
```json
{
  "success": true,
  "data": {
    "session_id": "string",
    "generated_text": "string",
    "prompt_tokens": 32,
    "generated_tokens": 128,
    "cache_hit": true,
    "mode": "prefill | decode",
    "ttft_ms": 120,
    "total_latency_ms": 820
  }
}
```

强制行为规则（表格）
| 场景 | 行为 |
|---|---|
| 无 session_id | 自动创建 session |
| 首次请求 | prefill |
| 已存在 session | decode（增量输入） |
| cache 不存在 | fallback 执行 prefill |
| session 过期 | 重建 session |

## 五、Python Backend 设计（Tech Spec）

### 5.1 SessionState
```python
class SessionState:
    def __init__(self):
        self.past_key_values = None
        self.input_ids = None
        self.attention_mask = None
        self.token_count = 0
        self.last_access = time.time()
```
存储容器：SESSION_STORE = dict()

约束：
- KV Cache 保存在 Python 内存（不序列化为 JSON），按 session_id 索引
- Go 侧只传 session_id；禁止传递 tensors/浮点数组

### 5.2 核心接口定义
1) prefill(session_id, prompt, params)
- tokenize(prompt)
- 单次或批量 forward（use_cache=True）
- 保存完整 past_key_values（全部 Transformer 层）
- 返回生成文本及元数据（prompt_tokens、generated_tokens、ttft 等）

2) decode(session_id, new_prompt, params)
- 仅将新增 tokens 作为输入（incremental）
- 调用 model(..., past_key_values=session.past_key_values, use_cache=True)
- 按 token 逐步生成（for step in range(max_new_tokens): sample logits -> next_token -> append）
- 每步更新 session.past_key_values 与 token_count
- 支持 sampling（temperature + top_p + top_k）、EOS 检测、max_new_tokens 限制
- 保证 cache hit 情况下总耗时下降（目标 ≥30% 二轮 TTFT 降低）

### 5.3 正确生成伪代码
```python
outputs = model(
    input_ids=new_tokens,
    past_key_values=session.past_key_values,
    use_cache=True
)
next_token = sample(outputs.logits)
session.past_key_values = outputs.past_key_values
```
循环直到 EOS 或 max_new_tokens。

### 5.4 强制约束（表格）
| 项 | 要求 |
|---|---|
| KV Cache | 使用全部层的 past_key_values |
| JSON tensor | 禁止 |
| cache 存储 | Python 内存（按 session_id） |
| Go 传输 | 仅 session_id 与控制参数 |

## 六、Go 侧改造（Tech Spec）

### 6.1 Session 元数据
```go
type Session struct {
    SessionID   string
    CreatedAt   time.Time
    UpdatedAt   time.Time
    Model       string
    TokenCount  int
}
```

### 6.2 Session Manager 职责
- CreateSession / GetSession / UpdateSession / DeleteSession
- TTL 清理、LRU 淘汰（防止内存爆炸）
- 并发保护（锁或并发安全容器）
- 不持有 KV tensor，仅维护 metadata 与路由信息

### 6.3 BackendRequest（与 Python 交互）
```go
type BackendRequest struct {
    SessionID     string
    Prompt        string
    MaxNewTokens  int
    Temperature   float64
    TopP          float64
}
```
禁止行为：不允许传 KV tensor、不允许 Go 存 KV、不允许传 float32[][] 等 tensor 形式。

## 七、核心算法（Must-have）
- Autoregressive incremental decoding loop（逐 token）
- Sampling：temperature + top_p（可含 top_k）
- EOS 停止判断
- max_new_tokens 强制上限
可选加分：repetition penalty、stop words、beam search（非必需）

---

## 八、性能与正确性指标（验收标准）

延迟
| 指标 | 要求 |
|---|---|
| 首轮 TTFT | baseline |
| 二轮 TTFT | 相比首轮至少 ↓30% |
| 总延迟 | 明显下降（以 TTFT 与总耗时为准） |

正确性
- cache 与 no-cache 输出差异在可接受范围
- 无 crash、无错 token（tokenization/offset 保证）
并发
- ≥50 并发 session，不串、不污染
内存
- 无明显泄漏；cache 可回收（TTL + LRU）

## 十二、风险与缓解
1. KV Cache 结构差异（HF transformers 版本差异）  
   缓解：锁定 transformers 版本并在 CI 中校验 outputs.past_key_values 结构。

2. 内存爆炸（session 数量过多导致）  
   缓解：session TTL + LRU 淘汰 + 限制单 session 最大缓存大小。

3. session 串数据/并发竞争  
   缓解：严格隔离 session 读写、key 校验与并发锁策略、端到端测试覆盖。
