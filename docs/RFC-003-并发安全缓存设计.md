# RFC-003: 并发安全的缓存设计

## 元信息
- **RFC 编号**：003
- **标题**：并发安全的缓存设计
- **状态**：已实现
- **创建日期**：2026-03-27
- **作者**：Llama-Go Team

## 背景

### 并发场景
AI 推理服务需要处理：
- 多用户同时请求
- 同一用户的并发请求
- 缓存的频繁读写

### 挑战
- **数据竞争**：多个 goroutine 同时读写 map
- **性能瓶颈**：锁竞争影响吞吐量
- **内存安全**：避免 panic 和数据损坏

## 设计方案

### 核心数据结构
```go
type KVCache struct {
    cache map[string][]*CacheEntry
    mu    sync.RWMutex
}
```

### 读写锁机制

**读锁（RLock）**：
- 允许多个 goroutine 并发读
- 适用于高读场景
- 不阻塞其他读操作

**写锁（Lock）**：
- 独占访问
- 阻塞所有读写操作
- 保证数据一致性

## 实现细节

### 读操作（并发友好）
```go
func (k *KVCache) Get(sessionID string) []*CacheEntry {
    k.mu.RLock()
    defer k.mu.RUnlock()
    return k.cache[sessionID]
}
```

**特点**：
- 多个请求可同时读取
- 不影响其他读操作
- 性能优秀

### 写操作（互斥保护）
```go
func (k *KVCache) Set(sessionID string, entry *CacheEntry) {
    k.mu.Lock()
    defer k.mu.Unlock()
    k.cache[sessionID] = append(k.cache[sessionID], entry)
}
```

**特点**：
- 独占访问，保证安全
- 阻塞其他读写
- 写操作较少，影响可控

### 删除操作
```go
func (k *KVCache) Clear(sessionID string) {
    k.mu.Lock()
    defer k.mu.Unlock()
    delete(k.cache, sessionID)
}
```

## 性能分析

### 读写比例
典型 AI 对话场景：
- 读操作：80%（获取历史缓存）
- 写操作：20%（存储新缓存）

### RWMutex 优势
| 场景 | Mutex | RWMutex | 提升 |
|------|-------|---------|------|
| 高读低写 | 串行 | 并发读 | 5-10x |
| 均衡读写 | 串行 | 部分并发 | 2-3x |
| 高写低读 | 串行 | 略优 | 1.2x |

### 并发测试
```go
// 100 个并发读
for i := 0; i < 100; i++ {
    go cache.Get("session1")
}
// 无竞争，性能线性扩展
```

## 替代方案对比

### 方案 1：sync.Map
**优点**：
- 官方并发安全 map
- 无需手动加锁

**缺点**：
- 适合读多写少且 key 稳定的场景
- 复杂数据结构不友好
- 性能不如 RWMutex + map

**结论**：不适合本场景

### 方案 2：分片锁（Sharded Lock）
**优点**：
- 降低锁竞争
- 更高并发度

**缺点**：
- 实现复杂
- 当前规模不需要

**结论**：过度设计，暂不采用

### 方案 3：无锁数据结构
**优点**：
- 最高性能

**缺点**：
- 实现极其复杂
- 维护成本高

**结论**：不适合

## 扩展性考虑

### 内存限制
**问题**：缓存无限增长

**解决方案**：
```go
const MaxCacheSize = 1000

func (k *KVCache) Set(sessionID string, entry *CacheEntry) {
    k.mu.Lock()
    defer k.mu.Unlock()

    entries := k.cache[sessionID]
    if len(entries) >= MaxCacheSize {
        entries = entries[1:] // 删除最旧的
    }
    k.cache[sessionID] = append(entries, entry)
}
```

### 分布式缓存
**当前**：单机内存缓存

**升级路径**：
1. Redis 集中式缓存
2. 一致性哈希分片
3. 缓存预热和失效策略

## 测试策略

### 并发安全测试
```go
func TestConcurrentAccess(t *testing.T) {
    cache := NewKVCache()
    var wg sync.WaitGroup

    // 100 个并发写
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            cache.Set("test", &CacheEntry{})
        }(i)
    }

    // 100 个并发读
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            cache.Get("test")
        }()
    }

    wg.Wait()
}
```


## 最佳实践

1. **defer 释放锁**：防止 panic 导致死锁
2. **最小锁粒度**：只保护必要的临界区
3. **避免嵌套锁**：防止死锁
4. **读写分离**：优先使用 RLock

## 结论

RWMutex 是当前场景的最佳选择，平衡了实现复杂度和性能。未来可根据实际负载考虑分布式缓存方案。
