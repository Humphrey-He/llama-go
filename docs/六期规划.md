# 一、背景

## 1. 当前项目阶段

根据当前项目说明，llama-go 已完成以下核心建设：

* Go + Python 混合推理架构
* 标准 Go 工程化重构
* 多 Backend 抽象层
* OpenAI 兼容 API
* SSE 流式推理
* Session 管理（TTL + LRU）
* 模型注册表
* 单元测试与 CI/CD

当前系统已经具备：

* 多 backend 推理能力
* OpenAI 风格服务接口
* 多轮会话支撑能力
* 流式/非流式兼容能力

这意味着项目已经越过“Demo/MVP”阶段，进入**生产化建设阶段**。

---

## 2. 当前主要短板

虽然当前版本从功能上“可用于生产环境”，但从大厂生产标准看，仍存在明显缺口：

### （1）服务治理能力不足

当前尚未系统性建设：

* 超时控制
* 重试机制
* 熔断降级
* backend 健康治理
* 优雅退出

这会导致：

* backend 抖动时 API 长时间阻塞
* 流式连接难以管理
* 上线后故障恢复能力弱

---

### （2）可观测性不足

当前项目尚未形成完整 observability 体系：

* 缺少标准指标
* 缺少结构化日志
* 缺少 tracing
* 缺少 token 使用统计
* 缺少面向模型/backend/session 的核心监控维度

这会导致：

* 问题无法快速定位
* 性能瓶颈无法识别
* 成本和容量不可评估

---

### （3）安全和资源治理不足

当前未具备生产入口最基本的保护能力：

* 无 API Key 鉴权
* 无限流机制
* 无配额与 token 控制
* 无审计日志

这会导致：

* 服务易被滥用
* 无法做多租户或业务接入
* 资源与成本不可控

---

### （4）部署交付不完整

当前仍缺少：

* docker-compose 一键启动
* Kubernetes 标准部署模板
* 配置分层管理
* 部署/回滚/升级文档
* 容量测试和基准报告

这会导致：

* 交付成本高
* 上线流程不标准
* 无法支撑团队协作和运维接手

---

## 3. 本次任务定位

第六次开发不是单点补功能，而是建立：

> **一套完整的生产级运行体系，让 llama-go 具备稳定、可观测、安全、可交付、可评估的服务能力。**

# 二、目标

## 2.1 总体目标

在现有多 backend、OpenAI 兼容、流式推理基础上，完成**生产级服务治理与部署体系建设**。

---

## 2.2 核心目标

### 目标一：服务可靠性治理

建立从 HTTP 层到 backend 层的容错与恢复能力，确保服务在 backend 波动、网络异常、流式中断等场景下稳定运行。

### 目标二：可观测性体系

建立日志、指标、Tracing 三位一体的 observability 体系，支持性能分析、故障定位、容量评估与成本统计。

### 目标三：安全与资源控制

增加 API Key 鉴权、限流、token usage 统计和审计能力，支撑多业务方/多租户接入。

### 目标四：标准化部署与运维交付

提供本地和集群环境的标准部署方案，使项目具备一键启动、标准上线、灰度升级和运维接管能力。

### 目标五：性能基准与容量评估

建立标准 benchmark 流程，为后续上线规模、实例规格、限流阈值和扩容策略提供数据依据。

---

## 2.3 非目标

本轮明确不做：

* ❌ 多机分布式调度
* ❌ 多区域容灾
* ❌ 模型训练/微调
* ❌ 权限中心/用户中心
* ❌ 完整控制台 UI
* ❌ 复杂计费系统
* ❌ tool calling / agent orchestration

# 三、模块拆解

---

## 3.1 可靠性治理模块

### 模块目标

为 API 层、stream 层、backend 调用层增加超时、重试、熔断、健康探测与优雅退出能力。

### 新增/改造模块建议

```text
internal/reliability/
    timeout.go
    retry.go
    circuit_breaker.go
    health.go
```

### 功能拆解

#### 1）超时控制

分层设置：

* Server Read Timeout
* Server Write Timeout
* Request Context Timeout
* Backend Call Timeout
* Stream Idle Timeout
* Graceful Shutdown Timeout

#### 2）重试机制

仅允许对以下场景重试：

* backend 建连失败
* backend 短暂 5xx
* backend 超时前未开始流式写出

禁止重试：

* SSE 已开始输出
* session 已进入不可幂等更新阶段
* 客户端主动断开

#### 3）熔断器

对 backend 维度建立熔断器：

* failure threshold
* half-open 状态探测
* open timeout
* 恢复窗口

#### 4）健康检查增强

现有 `/healthz` 和 `/readyz` 升级为：

* liveness：进程存活
* readiness：服务可接流量
* backend readiness：依赖 backend 是否健康
* model readiness：模型注册与可路由状态

#### 5）优雅退出

服务收到退出信号时：

* 停止接新请求
* 等待现有非流式请求完成
* 尝试结束现有 stream
* flush 最终日志和指标

# 五、风险

## 风险 1：可靠性逻辑引入额外复杂度

问题：

* timeout/retry/breaker 增加链路复杂度
* 容易出现误重试、误熔断

应对：

* 重试范围严格限定
* breaker 默认保守配置
* 所有治理动作有独立指标和日志

---

## 风险 2：流式场景治理难度高

问题：

* SSE 一旦开始写出，请求不可简单回滚
* 客户端断连、backend 阻塞、空闲流等问题更复杂

应对：

* stream 单独超时和空闲检测
* 明确 stream 与 non-stream 的重试边界
* 优雅退出时优先处理 stream draining

---

## 风险 3：指标过多导致维护成本上升

问题：

* 指标标签爆炸
* 高基数标签影响 Prometheus 性能

应对：

* 禁止将 request_id、session_id 作为 metrics label
* label 仅使用低基数字段，如 route/model/backend/status

---

## 风险 4：安全能力做得不完整导致“看似安全”

问题：

* 只加 API key，但无限流/无审计，效果有限

应对：

* API key、限流、审计至少一起交付最小闭环
* 明确哪些能力本轮必须可用，哪些仅预留

---

## 风险 5：Kubernetes 模板与实际运行不匹配

问题：

* YAML 能部署但不可运维
* 探针、资源、优雅退出未验证

应对：

* 基于真实压测与滚动发布验证模板
* deployment 文档与 benchmark 结论联动更新

---

## 风险 6：容量评估结论失真

问题：

* benchmark 场景过于理想化
* 未覆盖 stream、长文本、异常场景

应对：

* 设计多组真实场景
* 输出报告时明确硬件、模型、backend 前提
* 不给绝对结论，只给约束条件下建议值

---
