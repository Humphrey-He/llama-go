# 一、项目背景与迭代定位
## 1.1 当前项目所处阶段
llama-go 已完成MVP到功能完备的核心跃迁，已落地的核心能力包括：
- Go + Python 混合推理异构架构
- 标准Go工程化重构与分层设计
- 多Backend推理抽象层与模型注册表
- OpenAI 全兼容API接口（流式/非流式）
- SSE 长链接流式推理全链路支持
- Session 生命周期管理（TTL+LRU双策略）
- 单元测试体系与CI/CD流水线搭建

**架构师视角**：当前系统已完成核心业务逻辑闭环，具备基础的服务接入能力，但未达到大厂生产环境准入标准，缺少生产级服务必须的容错、隔离、可观测、安全管控、标准化交付能力。
**算法工程师视角**：当前系统已完成推理功能的全场景覆盖，但推理链路的性能、稳定性、可复现性不可控，缺少推理全流程的指标观测、异常定位、性能基准能力，无法支撑规模化推理业务的成本与效果管控。

基于此，项目已正式越过「Demo/MVP验证阶段」，进入**生产化落地攻坚阶段**，本次迭代是项目上线前的核心准入门槛建设。

## 1.2 当前生产级核心短板
### （1）服务可靠性治理能力缺失（架构师核心关注）
当前系统未建立全链路容错与故障隔离体系，缺少分层超时控制、幂等重试、熔断降级、Backend全生命周期健康治理、服务优雅退出能力。
**风险影响**：Backend抖动时API请求长时间阻塞、流式连接泄漏、服务发布时流量损失、故障恢复能力弱，无法满足生产环境99.9%以上的可用性要求。

### （2）可观测性体系不完整（架构师+算法工程师双核心关注）
当前系统未建立日志、指标、链路追踪三位一体的可观测性体系，缺少标准化核心指标、统一结构化日志、全链路分布式追踪、精准Token用量统计、多维度监控能力。
**架构师视角风险**：线上故障无法快速根因定位、性能瓶颈无法精准识别、服务容量无法科学评估。
**算法工程师视角风险**：推理性能波动无法归因、Token统计与模型侧不一致、长上下文推理的效果与性能无法量化、模型选型无数据支撑。

### （3）安全与资源治理能力不足（架构师核心关注）
当前系统未建立生产入口级安全防护体系，缺少API Key鉴权、全维度限流、多租户Token配额管控、全接口审计日志能力。
**风险影响**：服务易被恶意滥用、无法支撑多业务方接入、资源与推理成本不可控、无合规审计能力，无法满足企业级服务安全要求。

### （4）标准化部署与运维交付能力缺失（架构师+SRE双核心关注）
当前系统未建立标准化部署运维体系，缺少docker-compose一键编排、生产级K8s部署模板、分层配置管理、标准化运维流程、容量测试报告。
**风险影响**：服务交付成本高、上线流程不规范、无法支撑运维团队接管、无法规模化部署与弹性扩缩容。

### （5）推理性能基准与容量评估体系空白（算法工程师核心关注）
当前系统未建立标准化推理性能基准测试体系，缺少全场景测试用例、行业对齐的指标统计口径、服务容量科学评估方法、生产配置调优的数据依据。
**风险影响**：生产部署的资源规格、限流阈值、扩容策略无数据支撑，易出现资源浪费或服务过载，无法保障推理服务的性能与成本平衡。

## 1.3 本次迭代核心定位
第六期开发不是单点功能补全，而是**以大厂生产准入标准为目标，搭建一套完整的生产级运行体系**，让llama-go从「功能完备的LLM服务」，升级为「可稳定上线、可业务接入、可运维托管、可量化评估的企业级LLM平台底座」。

一句话定义：前五次开发解决了「系统能不能跑」的问题，本次开发解决「系统能不能在生产环境放心跑」的问题。

# 二、迭代目标与边界
## 2.1 总体目标
在现有多Backend兼容、OpenAI API兼容、全场景推理能力的基础上，完成**生产级服务治理、可观测性、安全管控、标准化部署、性能基准五大体系建设**，达成大厂生产环境服务准入标准。

## 2.2 核心目标（分角色）
### 架构师视角核心目标
1. **服务可靠性治理**：建立从HTTP入口到Backend调用的全链路容错与故障隔离体系，保障服务在异常场景下的稳定运行，可用性达到99.9%以上。
2. **可观测性体系落地**：建立日志、指标、链路追踪三位一体的标准化可观测性体系，支持线上故障分钟级定位、性能瓶颈精准识别。
3. **安全与资源管控闭环**：完成API鉴权、限流、并发管控、审计日志的最小闭环建设，支撑多租户安全接入，实现服务资源与成本可控。
4. **标准化部署交付**：提供本地与集群环境的标准化部署方案，实现服务一键启动、标准上线、灰度升级、无损上下线，满足运维接管要求。

### 算法工程师视角核心目标
1. **推理链路可靠性保障**：实现全推理场景的容错与异常恢复，保障推理结果一致性，杜绝治理逻辑对推理效果的影响。
2. **推理指标体系标准化**：建立与行业标准、模型侧完全对齐的推理核心指标统计口径，实现推理性能、Token用量的精准统计。
3. **性能基准与容量评估**：搭建标准化推理基准测试体系，输出生产环境部署的资源规格、并发阈值、配置参数的量化建议。
4. **模型/Backend生命周期治理**：建立模型与Backend的健康探测、状态管理、故障隔离机制，保障多模型场景下的推理稳定性。

## 2.3 迭代边界（明确非目标）
本轮迭代严格控制范围，**明确不做**以下内容，避免需求蔓延：
- ❌ 多机分布式推理调度与集群管理
- ❌ 多区域跨机房容灾部署
- ❌ 模型训练/微调/量化相关能力
- ❌ 完整的用户中心/权限管理体系
- ❌ 可视化控制台UI界面
- ❌ 复杂的计费与清结算系统
- ❌ Tool Calling/Agent编排相关能力
- ❌ 模型内容安全与审核过滤体系

以上内容纳入后续迭代规划，本轮仅预留必要的扩展接口。

# 三、核心架构设计原则
为保障本次迭代的架构一致性与生产级质量，所有模块开发必须严格遵循以下设计原则。

## 3.1 架构师视角架构原则
1. **非侵入式设计**：所有治理逻辑通过中间件/插件化方式实现，完全不侵入核心推理业务逻辑，保障核心代码稳定性。
2. **故障隔离原则**：实现请求级、租户级、模型级、Backend级的多级故障隔离，单个实例故障不影响全局服务可用性。
3. **可配置化原则**：所有治理规则、阈值参数均支持通过配置文件/环境变量/ConfigMap动态调整，无需重启服务。
4. **可观测性先行**：所有治理动作、业务流程、异常事件必须有对应的日志、指标、链路追踪可追溯，无观测盲点。
5. **最小权限原则**：安全管控遵循最小权限原则，API Key仅开放必要的接口与模型访问权限，杜绝过度授权。
6. **完全兼容性原则**：100%兼容已有的OpenAI API接口规范，无破坏性变更，不影响现有业务方接入。
7. **运维友好原则**：所有部署配置、监控指标、故障排查手册均符合SRE运维规范，支持标准化运维操作与自动化管控。

## 3.2 算法工程师视角架构原则
1. **推理一致性优先**：所有治理逻辑不得改变模型推理的输入输出结果，不得影响推理上下文的一致性，保障推理效果可复现。
2. **指标零偏差原则**：推理核心指标、Token用量统计口径必须与模型分词器、Backend侧完全对齐，统计误差控制在0.1%以内。
3. **性能无损原则**：所有新增治理逻辑引入的端到端额外延迟必须控制在5ms以内，不得对推理性能产生可感知的影响。
4. **全场景覆盖原则**：所有能力必须覆盖流式/非流式、短文本/长上下文、单轮/多轮会话的全推理场景，无功能盲区。
5. **异常可复现原则**：所有推理异常事件必须记录完整的上下文信息（不含用户隐私数据），保障异常场景可复现与根因定位。

# 四、模块拆解与详细需求
本次迭代共拆解为6大核心模块，优先级定义：P0（阻塞上线，必须100%完成）、P1（高优，本轮必须完成）、P2（中优，本轮预留接口）

## 4.1 可靠性治理模块
### 模块定位
建立全链路容错与故障隔离体系，保障服务在各类异常场景下的稳定运行，是生产环境可用性的核心基础。
### 代码目录规范
```text
internal/reliability/
    timeout.go          # 全链路超时控制
    retry.go            # 幂等重试机制
    circuit_breaker.go  # 熔断器
    health.go           # 健康检查
    graceful.go         # 优雅退出
    middleware.go       # HTTP层治理中间件
```
### 详细需求拆解
| 功能点 | 优先级 | 详细需求 |
|--------|--------|----------|
| 全链路分层超时控制 | P0 | 1. 实现6层超时管控：Server Read/Write超时、Request上下文超时、Backend调用超时、Stream空闲超时、优雅退出超时<br>2. 超时时间支持按模型/接口/租户维度差异化配置<br>3. 超时事件必须有完整的日志、指标记录，明确超时层级与原因 |
| 优雅退出与无损上下线 | P0 | 1. 服务收到SIGINT/SIGTERM信号时，立即停止接收新请求<br>2. 等待现有非流式请求处理完成，超时强制终止<br>3. 流式请求发送结束标识，优雅关闭连接<br>4. 退出前Flush所有日志、指标、Session数据<br>5. 支持K8s滚动更新时的流量无损切换 |
| 增强型健康检查 | P0 | 1. 拆分3类健康检查接口，严格遵循K8s探针规范：<br>   - /healthz：进程存活探针，仅返回进程状态<br>   - /readyz：服务就绪探针，检查所有依赖是否正常，异常时返回非200状态码<br>   - /livez：后端就绪探针，检查Backend/模型的可服务状态<br>2. 健康检查结果支持指标化暴露，可被监控系统采集 |
| 幂等重试机制 | P1 | 1. 仅允许对幂等场景重试：Backend建连失败、未开始流式输出的5xx错误、超时前未返回首包的请求<br>2. 严格禁止重试场景：已开始SSE流式输出、Session已更新、客户端主动断开、4xx类错误<br>3. 支持重试次数、重试间隔、最大重试时长配置，重试请求必须透传Trace上下文<br>4. 重试事件有独立的指标与日志记录 |
| Backend维度熔断器 | P1 | 1. 基于Backend实例维度实现熔断器模式，支持故障阈值、半开探测窗口、恢复阈值配置<br>2. 熔断器状态：关闭→打开→半开→关闭，状态变更有事件日志与指标记录<br>3. 熔断器打开时，自动将流量切换到健康Backend实例，实现故障隔离<br>4. 支持手动熔断/恢复Backend实例，便于线上应急处理 |

## 4.2 可观测性模块
### 模块定位
建立日志、指标、链路追踪三位一体的可观测性体系，实现服务全生命周期的可监控、可排查、可评估。
### 代码目录规范
```text
internal/observability/
    metrics.go      # Prometheus指标定义与注册
    logger.go       # 结构化日志封装
    tracing.go      # OpenTelemetry链路追踪
    middleware.go   # HTTP层可观测中间件
    consts.go       # 指标/标签常量定义
```
### 详细需求拆解
| 功能点 | 优先级 | 详细需求 |
|--------|--------|----------|
| 统一结构化日志 | P0 | 1. 全链路统一使用JSON格式结构化日志，禁止零散的fmt打印<br>2. 强制必填字段：时间戳、日志级别、request_id、trace_id、session_id、api_key_id、接口路由、模型、Backend、状态码、延迟<br>3. 安全约束：禁止明文打印API Key、禁止完整打印用户Prompt/Completion内容<br>4. 错误日志必须包含错误类型、错误码、根因信息 |
| Prometheus核心指标体系 | P0 | 1. 实现/metrics接口，支持Prometheus标准格式指标暴露<br>2. 指标覆盖6大维度：HTTP服务、推理核心、Backend状态、鉴权限流、Session管理、资源治理<br>3. 严格禁止高基数标签：request_id、session_id不得作为指标标签<br>4. 指标类型严格遵循Prometheus规范：Counter/Histogram/Gauge分类清晰 |
| 全链路上下文透传 | P0 | 1. 实现全链路请求上下文统一注入与透传，核心字段：request_id、trace_id、session_id、api_key_id、model、backend<br>2. 上下文必须在HTTP层、鉴权层、推理层、Backend调用层全链路透传<br>3. 日志、指标、链路追踪必须统一使用上下文内的字段，保证数据一致性 |
| OpenTelemetry分布式追踪 | P1 | 1. 接入OpenTelemetry SDK，实现全链路分布式追踪<br>2. 核心Span覆盖：HTTP请求、鉴权校验、限流检查、Session查询、模型路由、Backend推理、流式写入<br>3. Span必须包含核心属性：request_id、model、backend、stream标识、api_key_id、重试次数、熔断器状态<br>4. 支持Jaeger/Zipkin等主流平台对接，支持采样率配置 |
| Token用量精准统计 | P1 | 1. 实现与模型分词器完全对齐的Prompt/Completion Token统计，统计误差<0.1%<br>2. 支持按API Key、模型、Backend维度的Token用量累计统计<br>3. 统计结果必须写入日志、指标，支持用量查询与成本核算<br>4. 流式场景下支持Token逐包累计统计，最终结果与非流式场景一致 |

## 4.3 安全与访问控制模块
### 模块定位
建立生产环境入口级安全防护体系，实现身份认证、访问控制、合规审计的最小闭环，支撑多租户安全接入。
### 代码目录规范
```text
internal/auth/
    apikey.go       # API Key鉴权逻辑
    middleware.go   # 鉴权中间件
    consts.go       # 错误码与常量定义

internal/ratelimit/
    limiter.go      # 限流器实现
    quota.go        # 配额管理
    middleware.go   # 限流中间件
```
### 详细需求拆解
| 功能点 | 优先级 | 详细需求 |
|--------|--------|----------|

## 5.1 管理与观测接口规范
### 1. 存活探针接口 GET `/healthz`
- 用途：K8s liveness探针，判断进程是否存活
- 响应格式：
```json
{
  "status": "ok",
  "timestamp": "2026-03-27T10:00:00Z",
  "version": "v1.0.0"
}
```
- 响应码：正常返回200，进程异常返回503

### 2. 就绪探针接口 GET `/readyz`
- 用途：K8s readiness探针，判断服务是否可接收流量
- 响应格式：
```json
{
  "status": "ready",
  "timestamp": "2026-03-27T10:00:00Z",
  "checks": {
    "backend": "ok",
    "model_registry": "ok",
    "session_store": "ok",
    "auth_system": "ok"
  }
}
```
- 响应码：所有检查项正常返回200，任意一项异常返回503

### 3. 后端就绪接口 GET `/livez`
- 用途：判断Backend与模型的可服务状态
- 响应格式：同`/readyz`，仅返回Backend与模型相关检查项
- 响应码：正常返回200，异常返回503

### 4. 指标接口 GET `/metrics`
- 用途：Prometheus指标拉取
- 响应格式：Prometheus文本格式
- 响应码：正常返回200，异常返回500
