# RFC-002: Go + Python 混合架构设计

## 元信息
- **RFC 编号**：002
- **标题**：Go + Python 混合架构设计
- **状态**：已实现
- **创建日期**：2026-03-27
- **作者**：Llama-Go Team

## 背景

### 技术选型挑战
AI 推理服务需要同时满足：
1. **高并发处理**：处理大量用户请求
2. **AI 生态支持**：使用成熟的深度学习框架
3. **开发效率**：快速迭代和部署

单一语言难以兼顾所有需求。

## 设计方案

### 架构选择：微服务分离

```
┌─────────────┐      HTTP      ┌──────────────┐
│  Go Server  │ ──────────────▶│ Python Server│
│  (API 层)   │                │  (推理层)     │
└─────────────┘                └──────────────┘
```

### 职责划分

**Go 服务职责**：
- HTTP 请求处理
- 并发控制
- KV Cache 管理
- 会话管理
- 负载均衡

**Python 服务职责**：
- 模型加载
- 推理计算
- KV Cache 提取
- 结果返回

## 技术决策

### 为什么不用纯 Python？
**问题**：
- GIL（全局解释器锁）限制并发
- 内存管理效率低
- 缓存操作性能差

**对比**：
| 指标 | Python | Go |
|------|--------|-----|
| 并发模型 | 多进程/异步 | Goroutine |
| 内存管理 | 引用计数 | GC 优化 |
| 并发性能 | 中等 | 优秀 |

### 为什么不用纯 Go？
**问题**：
- AI 生态不成熟
- 缺少成熟的深度学习框架
- 模型资源少

**对比**：
| 指标 | Go | Python |
|------|-----|--------|
| AI 框架 | 有限 | 丰富 |
| 预训练模型 | 少 | 海量 |
| 社区支持 | 弱 | 强 |

## 通信协议

### HTTP REST API
**选择理由**：
- 简单易用
- 语言无关
- 易于调试

**接口定义**：
```json
POST /infer
Request:
{
  "prompt": "string",
  "kv_cache": "object|null"
}

Response:
{
  "text": "string",
  "kv": {"keys": [], "vals": []},
  "token_num": "int"
}
```

### 数据序列化
- **格式**：JSON
- **优点**：可读性强，调试方便
- **缺点**：性能略低
- **优化方向**：可改用 Protocol Buffers

## 部署方案

### Docker Compose 编排
```yaml
services:
  go-server:
    ports: ["8080:8080"]
    depends_on: [python-backend]

  python-backend:
    ports: ["8000:8000"]
```

### 服务发现
- 当前：硬编码 `localhost:8000`
- 改进：环境变量配置
- 生产：服务注册中心（Consul/Etcd）

## 性能优化

### 连接池
```go
var httpClient = &http.Client{
    Timeout: 30 * time.Second,
    Transport: &http.Transport{
        MaxIdleConns: 100,
        MaxIdleConnsPerHost: 10,
    },
}
```

### 批处理
- 当前：单请求推理
- 优化：批量推理（提升 GPU 利用率）

## 风险与挑战

### 网络延迟
- **问题**：跨服务调用增加延迟
- **缓解**：本地部署，减少网络跳转

### 故障隔离
- **问题**：Python 服务崩溃影响整体
- **缓解**：健康检查 + 自动重启

### 数据一致性
- **问题**：缓存与推理状态可能不同步
- **保障**：事务性更新，失败回滚
