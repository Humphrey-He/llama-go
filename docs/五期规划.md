# 一、背景（Why）

## 1. 当前系统状态

基于仓库当前实现，系统存在以下结构性问题：

### （1）协议层非标准

当前 API：

* `/api/generate`
* `/api/clear`

问题：

* 非 OpenAI 标准
* 无 chat message 抽象
* 无 usage / finish_reason
* 无 streaming
* 无模型管理

👉 无法接入：

* OpenAI SDK
* LangChain / LlamaIndex
* Agent 系统
* 前端 UI

---

### （2）Backend 强耦合

当前实现：

* Go 直接调用 Python HTTP `/infer`
* 请求/响应结构写死

问题：

* 无法扩展到 llama.cpp / vLLM
* 无法做多模型调度
* 无法做 fallback / routing

---

### （3）系统定位模糊

README 声称：

> Go 本地推理框架（GGUF）

实际：

* 推理在 Python Transformers
* Go 只是 API 层

👉 架构不清晰，后续扩展成本极高

---

## 2. 第四次开发后的前提

假设 Task-4 已完成：

* ✅ KV Cache 真正生效
* ✅ 支持多轮对话
* ✅ 推理闭环成立

👉 当前瓶颈从“算法正确性”转为：

> **系统是否具备平台能力**

---

## 3. 本次任务定位

第五次开发不是优化，而是**架构升级**：

> **从“单一推理 Demo” → “标准化推理服务网关”**

# 二、目标（What）

构建一个**可对接生态、可扩展 backend、支持流式输出的 LLM 推理网关**

---

# 三、接口定义（API Spec）

## 3.1 Chat Completions（核心接口）

### POST `/v1/chat/completions`

---

### 请求

```json
{
  "model": "tinyllama-chat",
  "messages": [
    {"role": "system", "content": "You are helpful."},
    {"role": "user", "content": "解释Transformer"}
  ],
  "temperature": 0.7,
  "top_p": 0.9,
  "max_tokens": 256,
  "stream": true
}
```

---

### 非流式响应

```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "created": 1710000000,
  "model": "tinyllama-chat",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Transformer 是..."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 32,
    "completion_tokens": 128,
    "total_tokens": 160
  }
}
```

---

### 流式响应（SSE）

Header：

```
Content-Type: text/event-stream
```

事件：

```text
data: { "choices": [{ "delta": { "content": "Trans" } }] }

data: { "choices": [{ "delta": { "content": "former" } }] }

data: [DONE]
```

# 七、风险（Risks）

---

## 风险 1：流式输出实现错误

问题：

* 一次性缓冲输出
* 无 flush

解决：

* 强制每 chunk flush
* 单测验证 chunk 数量

---

## 风险 2：backend 接口设计不合理

问题：

* 后续无法接 vLLM / llama.cpp

解决：

* 保持最小抽象
* 不暴露模型内部结构

---

## 风险 3：协议不兼容 OpenAI

问题：

* SDK 无法调用

解决：

* 严格对齐字段
* 用官方 SDK 测试

---

## 风险 4：stream 无法 cancel

问题：

* 客户端断开仍在推理

解决：

* context cancel 传递到 backend

---

## 风险 5：代码耦合回退

问题：

* API 又直接调用 Python

解决：

* 强制 code review 规则：

  * ❌ 禁止直接 HTTP 调 Python
  * ✅ 必须通过 interface

## 结论（简短）
总体可行。设计思路与目标清晰，按大厂 Tech Spec 标准可落地；但需补齐若干细节（接口契约、并发/资源边界、测试与回归策略、运维保障）以降低实现风险并确保可验证交付。

## 关键验收条件（必须在 Task-5 完成前明确）
- OpenAI 兼容接口行为逐字段校验用例（含字段缺失/异常/限流/错误码）。
- Backend 抽象接口的同步/流式契约（超时、取消、重试、错误类型、并发语义）。
- 模型注册表规范（能力字段、版本、health-check、优先级）。
- 流式 SSE 语义（每 chunk JSON schema、DONE 事件、flush 频率、心跳与 reconnect 指导）。
- CI 集成测试（adapter 单测、python backend stub 集成测试、SSE 端到端测试）。
- 性能指标与测量方法（TTFT 定义、目标值、测试脚本与负载点）。
- 安全/限流策略（单会话最大 tokens、并发推理上限、请求速率、资源隔离建议）。

## 主要问题与建议（按优先级）
1. 抽象接口细化（高）
   - 建议将 Generate/GenerateStream 返回错误结构化（error code, retriable bool, backend id）并定义语义化错误集（e.g., ErrContextExceeded, ErrBackendBusy）。
   - 明确 GenerateStream 的 backpressure：channel capacity、是否支持 pause/resume、最大未消费 chunk 数。

2. Context cancel 与资源回收（高）
   - 强制将 ctx 贯穿底层（包括 HTTP 客户端到 Python），并在 backend 侧实现可中断推理（短轮询/流式中间检查点或显式 cancel API）。
   - 明确断连清理策略（GC 延迟、最大保留时间、关闭信号路径）。

3. 并发与容量限制（高）
   - 设计并发配额与排队策略（per-model、per-session、per-backend）。
   - 指定拒绝策略（429 vs 503）与降级（返回 mock/错误/queued）。

4. Streaming 可靠性（中）
   - 明确定义 chunk 最小单元（token/byte），以及合并策略（多 token 合并发出阈值）。
   - 为客户端取消与网络抖动提供重连/续流策略（session id + resume offset）。

5. 后端扩展兼容性（中）
   - interface.go 要包含 BackendInfo.Capabilities（supports_stop_sequences, supports_token_streaming, max_context, concurrency）以便路由与能力判断。
   - python_backend.go 的适配层建议实现能力探测与健康检查接口。

6. 可观测性与埋点（中）
   - 明确埋点事件与字段（request_start, backend_call_start/end, token_stream_chunk, request_end）以及日志等级与隐私处理。
   - 指定 Prometheus 指标（active_requests, ttft_histogram, stream_chunk_rate, backend_errors_total）。

7. 测试策略（必须）
   - 单元：adapter、model registry、stream 模块。
   - 集成：mock backend 全链路（stream 与 non-stream），python backend CI 环境下稳定性测试。
   - 性能：TTFT 与吞吐测试脚本，包含 cancel/断连场景。

## 执行建议（里程碑与估算）
假设小团队（2 后端，1 qa），已完成 Task-4 基础：
- Phase A（1周）：细化接口契约、错误集、能力字段、测试矩阵；完成 PRD/接口 mock。  
- Phase B（2周）：实现 backend interface + mock backend + model registry + unit tests。  
- Phase C（2周）：实现 OpenAI adapter 与非流式 /v1/chat/completions 集成测试；兼容 /api/generate。  
- Phase D（2周）：实现 stream SSE、GenerateStream 管道、client-cancel 支持、流式集成测试与 TTFT 测试。  
- Phase E（1周）：/v1/models、usage 统计、文档与最终回归、性能跑测。  
总计：~8周（±25%），含测试与修复；并行可缩短至5–6周。

## 小而重要的改动点（速做）
- 在 interface.go 增加 GenerateStream(ctx, req) (chan StreamChunk, cancelFunc, error) 或明确取消路径。
- 在 model registry 中加入 health-check interval 与 last_seen 字段。
- 定义 SSE 心跳 event（e.g., data: { "type": "ping" }）避免长轮询超时。
- 为 OpenAI SDK 兼容准备标准错误码映射表。

## 风险复核（补充）
- 若 python backend 无法中断推理，cancel/资源回收无法实现（需后端改造或中间代理）。
- 大量并发流式连接对 Go 服务器 FD/内存/GC 有压力，需压力测试并设置连接池/限制。
