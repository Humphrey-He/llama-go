## 核心模块设计

### 1. API 路由层 (api/routes.go)
**职责**：处理 HTTP 请求，协调各模块

**核心接口**：
- `POST /api/generate`：文本生成
- `POST /api/clear`：清空缓存

**流程**：
1. 接收请求并验证
2. 调用 Python 推理服务
3. 存储 KV Cache
4. 返回结果

### 2. KV Cache 管理 (internal/kvcache)
**职责**：管理推理过程中的 Key-Value 缓存

**数据结构**：
```go
type CacheEntry struct {
    Keys  [][]float32  // Key 张量
    Vals  [][]float32  // Value 张量
    Token int          // Token 位置
}

type KVCache struct {
    cache map[string][]*CacheEntry  // sessionID -> 缓存列表
    mu    sync.RWMutex               // 读写锁
}
```

**并发安全**：
- 读操作：`RLock()` 允许多个并发读
- 写操作：`Lock()` 独占写入

### 3. Backend 客户端 (internal/backend)
**职责**：与 Python 推理服务通信

**请求格式**：
```json
{
  "prompt": "用户输入",
  "kv_cache": null
}
```

**响应格式**：
```json
{
  "text": "生成文本",
  "kv": {"keys": [...], "vals": [...]},
  "token_num": 10
}
```

### 4. Python 推理服务 (python/kv_server.py)
**职责**：模型加载与推理

**核心流程**：
1. 加载 TinyLlama 模型
2. 接收推理请求
3. Tokenize 输入
4. 模型前向传播（use_cache=True）
5. 提取 past_key_values
6. 返回生成结果 + KV Cache

## 数据流设计

### 请求流程
```
Client Request
    ↓
Gin Router (验证参数)
    ↓
Backend Client (HTTP 调用)
    ↓
Python FastAPI (推理)
    ↓
Model Inference (生成 + KV)
    ↓
返回结果
    ↓
KVCache.Set (存储缓存)
    ↓
Response to Client
```

### KV Cache 工作原理

**问题**：Transformer 每次生成 token 都要重新计算所有历史 token 的 Attention

**解决**：缓存已计算的 Key 和 Value，新 token 只需计算自己的 K/V

**效果**：
- 首次推理：O(n²) 复杂度
- 后续推理：O(n) 复杂度
- 性能提升：2-5倍（取决于序列长度）

## 并发模型

### Go 并发优势
```go
// 多个请求并发读取缓存
func (k *KVCache) Get(sessionID string) []*CacheEntry {
    k.mu.RLock()  // 读锁，允许并发
    defer k.mu.RUnlock()
    return k.cache[sessionID]
}
```

### 会话隔离
- 每个 sessionID 独立缓存
- 支持多用户并发对话
- 缓存互不干扰
